# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo actually is

This is **not** a WordPress codebase — it is a prompt library and CI deployment harness for an agentic WordPress development workflow. The repo contains no PHP source of its own; the "code" here is:

1. Markdown prompt templates in `prompts/{dev,qa,deploy}/` that humans paste into three separate **Claude.ai Projects** (Dev / QA / Deploy agents).
2. The system prompts that define each agent, in `docs/claude-project-system-prompts.md`.
3. GitHub Actions workflows that FTP-deploy whatever lands in `generated-code/` (and any other PHP) to a staging or production WordPress server.

When working in this repo, you are almost always editing **prompts, docs, or workflow YAML** — not application code. The PHP that this system produces lives elsewhere (in `generated-code/` after a Dev Agent run, and ultimately on the WordPress server).

## The three-agent flow (the architecture)

The system intentionally splits the WordPress development loop across three Claude Projects with distinct system prompts and prompt templates. Each prompt template in `prompts/` is paired with one agent's system prompt:

- **Dev Agent** (`prompts/dev/*.md`) — generates WordPress PHP. Its system prompt enforces non-negotiable rules: WordPress Coding Standards, hooks/filters only (no core edits), sanitize all input, escape all output, nonce verification, `current_user_can()` checks, ABSPATH guard, `wp_enqueue_*` for assets, and a unique function/class/hook prefix (default `techforce_`).
- **QA Agent** (`prompts/qa/*.md`) — reviews Dev Agent output against a strict security/quality/performance rubric and emits a structured verdict (`SAFE TO DEPLOY` / `FIX FIRST` / `DO NOT USE`).
- **Deploy Agent** (`prompts/deploy/*.md`) — runs pre-deployment checklists and rollback plans. Always recommends staging-first.

When editing or adding a prompt template under `prompts/`, keep it consistent with the matching agent's system prompt in `docs/claude-project-system-prompts.md` — the two are designed to work as a pair, and changing rules in one without the other will break the workflow.

The default prefix `techforce_` is the per-client knob. `docs/client-config-template.md` is the canonical place to override it when forking the repo for a new client; the Dev Agent system prompt must then be updated to match.

## Deployment pipeline

Branch model is `feature/* → staging → main`:

- Push to `staging` → `.github/workflows/deploy-staging.yml` runs `php -l` syntax check and FTP-deploys to the staging server.
- Push to `main` → `.github/workflows/deploy-production.yml` runs the same check and deploys to production, but is gated by the GitHub `production` environment (manual approval required in repo settings).

Both workflows **exclude** `prompts/**`, `docs/**`, `.github/**`, and `README.md` from the FTP upload — only the generated WordPress code is shipped. If you add a new top-level directory that should not deploy (e.g. tooling, fixtures), add it to the `exclude:` block in **both** workflow files.

FTP credentials are read from GitHub Secrets named `STAGING_FTP_*` and `PROD_FTP_*` (server, username, password, path). These names are referenced in `docs/client-config-template.md` and must match in any forked client repo.

The only automated check in CI is `php -l` (syntax). There is no linter, no PHPCS, no PHPUnit runner configured at the repo level — quality gating happens through the QA Agent before code is committed.

## Working in this repo

- There are no build, lint, or test commands at the repo level. `tests/` is a placeholder for PHPUnit files generated by the Dev Agent on request; no test runner is configured here.
- `generated-code/` is the drop zone for Dev Agent output, named `feature-name_YYYY-MM-DD.php`. Treat anything in here as untrusted until a QA Agent pass has been recorded.
- `wp-config.php`, `.env*`, `vendor/`, and `node_modules/` are gitignored — never commit them, and never add real credentials to `docs/client-config-template.md` (it is a template; secrets belong in GitHub Secrets).
- When changing an agent's behavior, update both the system prompt in `docs/claude-project-system-prompts.md` **and** any affected prompt templates under `prompts/`. The user-facing instructions in `docs/HOW-TO-USE.md` and the table in `README.md` should also be kept in sync if the agent set or folder layout changes.

## Known issues

No outstanding deploy-workflow issues at this time. The Phase 4 bugs originally documented here have been resolved — see **Resolved** below.

### Resolved

1. **`Check PHP syntax` step failed on PHP-less repos.** Both `deploy-staging.yml` and `deploy-production.yml` ran `find . -name "*.php" -exec php -l {} \; | grep -v "No syntax errors"`. `grep -v` exits with status 1 on empty input, so the step failed on a repo with zero PHP files. **Fix:** replaced with `find . -path './.git' -prune -o -name '*.php' -print0 | xargs -0 -r -n1 php -l` and added a `shivammathur/setup-php@v2` step pinning PHP 8.2. The `xargs -r` flag (no-run-if-empty) makes the step pass cleanly when no PHP files exist, while still failing the build on real syntax errors.

2. **`actions/checkout@v3` was deprecated.** Both workflows pinned to v3. **Fix:** bumped both to `actions/checkout@v4`.

3. **FTP step failed loudly when secrets were unset.** `STAGING_FTP_*` and `PROD_FTP_*` were referenced unconditionally, so any deploy reaching the FTP step would fail when secrets weren't configured. **Fix:** the FTP step is now gated on `if: env.FTP_SERVER != ''`, and a follow-up step emits a `::warning::` annotation when secrets are missing. **Note:** the secrets themselves still need to be set in repo Settings → Secrets to enable real FTP uploads — this fix handles the missing-secrets case gracefully, it does not configure them. Both workflows also gained a `workflow_dispatch:` trigger so they can be re-run on demand from the Actions UI.

4. **`secrets.*` is not available in step-level `if:` expressions.** The first attempt at fix #3 used `if: ${{ secrets.STAGING_FTP_SERVER != '' }}`, which GitHub Actions rejects with `Unrecognized named-value: 'secrets'`. Step-level `if:` only sees the `env`, `github`, `inputs`, `steps`, `runner`, `job`, and `matrix` contexts. **Fix:** bridge each FTP server secret into a job-level `env:` block (`FTP_SERVER: ${{ secrets.STAGING_FTP_SERVER }}` for staging, `FTP_SERVER: ${{ secrets.PROD_FTP_SERVER }}` for production — `env:` at job level *can* reference `secrets.*`), then check `env.FTP_SERVER` in the step-level conditions.

## Agent behavior rules

### When the user asks to build a WordPress feature

1. Before writing any code, confirm you understand:
   - What the feature does (in plain English)
   - Where it appears (frontend, admin, both)
   - Which user roles can use it
   - Any plugin dependencies (WooCommerce, ACF, etc.)

   If any of these are unclear, ask before writing code.

2. Apply the Dev Agent rules from `docs/claude-project-system-prompts.md` without being reminded:
   - WordPress Coding Standards
   - Hooks/filters only, never modify core
   - `sanitize_*` on all input
   - `esc_*` on all output
   - `wp_verify_nonce` on all forms/AJAX
   - `current_user_can()` on privileged actions
   - All functions/classes/hooks prefixed with `techforce_` (or whatever prefix is set in `docs/client-config-template.md`)
   - `wp_enqueue_scripts` for all JS/CSS
   - `ABSPATH` check at top of every PHP file
   - PHPDoc on every function

3. Write each generated file directly to `generated-code/` with the filename format `feature-name_YYYY-MM-DD.php`. Do not output code blocks for the user to copy — write the actual file.

4. After writing files, perform a self-QA pass using the rubric in `prompts/qa/security-review.md`. Output the verdict in this exact format:

   ```
   Self-QA Verdict
   Security: PASS / FAIL / CRITICAL
   Code Quality: PASS / NEEDS FIXES
   Performance: GOOD / ACCEPTABLE / POOR
   Overall: SAFE TO DEPLOY / FIX FIRST / DO NOT USE
   ```

   If the verdict is `FIX FIRST` or `DO NOT USE`, fix the issues yourself and re-run the self-QA. Do not return control to the user with failing code.

5. Once self-QA passes, summarize:
   - Files created (with paths)
   - Hooks/filters used
   - Plugins required
   - Suggested feature branch name (format: `feature/<short-description>`)
   - Suggested commit message

   Do not commit or push — that is a separate step the user triggers explicitly.

### When the user asks to review existing code

Use the rubric in `prompts/qa/security-review.md`, `prompts/qa/coding-standards-check.md`, and `prompts/qa/performance-check.md`. Output the same verdict format above plus a numbered issues list with exact fixes.

### When the user asks about deployment

Refer to `prompts/deploy/pre-deployment-checklist.md` and `prompts/deploy/rollback-plan.md`. Always require explicit confirmation before any action that pushes to `staging` or `main`. Never push to `main` without confirming a database backup exists.

### Hard rules (never override)

- Never write secrets (API keys, passwords, FTP credentials) into any file in this repo. Secrets go in GitHub Secrets only.
- Never modify WordPress core files.
- Never commit to `main` directly. Always go through `feature` → `staging` → `main`.
- Never claim "deployed" — deployment is handled by GitHub Actions, you only prepare the code and the commit.
